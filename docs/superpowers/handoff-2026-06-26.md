**背景**: OAuth 计划 Task 6 给出的 `app/api/sync/route.ts`、`app/api/repos/[id]/star/route.ts` 代码使用旧的 2-arg 同步签名（`syncRepos(db, {...})`、`setStarred(db, repoId, bool)`），但 turso 计划 Tasks 3/4 已将这些改成异步 3/4-arg 签名（含 `userId`）。

### 决策 2: `createTestDb()` 用临时文件而非 `:memory:`

**背景**: `@libsql/client` 本地 sqlite3 驱动的 `transaction()` 方法（`lib-esm/sqlite3.js:158`）在执行 `db.transaction()` 后会把连接置为 `null`，下次访问惰性执行 `new Database(this.#path, ...)`。对于 `":memory:"` URL 这等于新建一个空数据库，migrate 建的表和数据全部丢失。

**修复**: `lib/db/test-helpers.ts` 中 `createTestDb()` 使用 `file:${mkdtempSync(...)}/test.db`（临时目录文件）。每次调用仍然是独立隔离的新库，只是从内存换成临时文件，规避重连陷阱。

**状态**: 已修复并在 Task 3 commit 中入库。临时文件不自动清理（测试结束后残留），这是已知 minor 问题，暂不处理。

## 待确认/待决策的问题

### 安全问题（来自 commit ad84860 的自动安全扫描）

1. **`session.accessToken` 可能暴露给客户端**
   - GitHub OAuth token 存在 session 对象里，JWT session 策略会把它编码进 cookie，前端页面通过 `useSession()` 即可读取
   - 对于个人工具（单用户），这个风险可能可接受；如果客户端组件从不读取 `session.accessToken`，风险很小
   - **待决策**: 是否将 `accessToken` 从客户端可见的 session 中移除，只在 API 路由里通过 `await auth()` 服务端读取？

2. **JWT 策略不检查 token 吊销**
   - 一旦登录，JWT 在过期前不会被撤销。GitHub revoke 该 token 后，session 仍然有效到 JWT 过期
   - 对于个人工具可接受；如需要强制吊销，需切换到 database session 策略（需要 adapter）
   - **待决策**: 接受现状，还是加 adapter？
